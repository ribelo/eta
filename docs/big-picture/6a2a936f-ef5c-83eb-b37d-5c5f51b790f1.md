## 1. Executive summary

Eta should add HTTP/2 server support as a **small effect server**, not a framework. The right split is:

* `eta_http`: backend-neutral server request/response/error/observability surface.
* `eta_http_eio`: h2c/TCP ownership, Eio accept loop, per-connection HTTP/2 owner loop, TLS/ALPN later.

This matches the repository’s existing boundary: `eta_http` already owns request/response models, typed errors, body streams, trace context, observability helpers, and pure protocol helpers, while runtime transports belong in adapters such as `eta_http_eio`. 

The MVP should be **h2c prior-knowledge HTTP/2**. TLS/ALPN should be the next phase, not forced into the MVP, because the current TLS surface is client-oriented and `eta_http_eio` already treats ALPN and transport dispatch as adapter-owned concerns. 

The central invariant: **one fiber/domain owns each `H2.Server_connection.t` and all `H2.Reqd` / body reader/writer mutation**. Handler fibers may run Eta effects concurrently, but they communicate with the connection owner through bounded commands. They must not mutate raw HTTP/2 state directly.

Multicore should use `Eio.Net.run_server` with `additional_domains` by default when a domain manager is available, not an Eta-specific request work-stealing layer. Eio’s server API explicitly supports additional accept/handling domains and warns that handlers must access only thread-safe values when used this way. ([ocaml-multicore.github.io][1])

OpenTelemetry should be on by default in the design: server spans, request/response/error attributes, active connection/stream metrics, lifecycle events, and safe URL/header redaction. Eta already has tracer/meter leaves and HTTP client observability conventions that can be mirrored for server spans and metrics.

## 4. Internal architecture

### 4.1 Accept loop

`Eta_http_eio.Server.run_h2c` should:

1. Open/listen on the supplied address with `Eio.Net.listen`.
2. Run `Eio.Net.run_server`.
3. Pass `~additional_domains` according to `domain_policy`.
4. For each accepted flow, create a connection switch and run one h2c server connection.

Sketch:

```ocaml
let run_h2c ~sw ~net ~clock ?domain_manager
    ?(domain_policy = Recommended) ?stop
    ?(config = Config.default) ?runtime_factory ~addr handler =
  let socket =
    Eio.Net.listen ~sw ~reuse_addr:true ~backlog:config.backlog net addr
  in
  let additional_domains =
    match domain_policy, domain_manager with
    | Single_domain, _ | _, None -> None
    | Recommended, Some dm ->
        Some (dm, max 0 (Domain.recommended_domain_count () - 1))
    | Additional n, Some dm ->
        Some (dm, max 0 n)
  in
  Eio.Net.run_server socket
    ?additional_domains
    ~max_connections:config.max_connections
    ~on_error:(fun exn -> (* log/metric, do not crash accept loop *))
    (fun ~sw:conn_sw flow peer ->
      H2.Server_connection.run_h2c
        ~sw:conn_sw ~clock
        ~flow:(flow :> H2.Server_connection.flow)
        ~peer ~config ~runtime_factory handler)
```

Default `Recommended` should use Eio’s additional-domain support only when a domain manager is supplied. Without one, run single-domain. Eio’s documentation says `run_server` is single-domain by default and can take `additional_domains:(domain_mgr, domains)`; it also cautions that the handler must access only thread-safe values when multiple domains are used. ([ocaml-multicore.github.io][1])

### 4.2 Per-connection ownership

Each connection should have one owner state:

```ocaml
type command =
  | Ingress of { bytes : Bigstringaf.t; off : int; len : int; ack : unit Eio.Promise.u }
  | Ingress_eof
  | Wake_writer
  | Request_body_read of stream_id * body_read_resolver
  | Request_body_discard of stream_id * discard_policy * unit_resolver
  | Response_start of stream_id * response_start
  | Response_chunk of stream_id * bytes * write_resolver
  | Response_trailers of stream_id * Eta_http.Core.Header.t * unit_resolver
  | Response_close of stream_id * unit_resolver
  | Handler_failed of stream_id * Eta_http.Server.Error.t
  | Stream_cancelled of stream_id
  | Shutdown of shutdown
```

The owner fiber is the only place that calls:

```ocaml
H2.Server_connection.read
H2.Server_connection.read_eof
H2.Server_connection.next_write_operation
H2.Server_connection.report_write_result
H2.Server_connection.yield_writer
H2.Reqd.respond_with_string
H2.Reqd.respond_with_streaming
H2.Reqd.report_exn
H2.Reqd.schedule_trailers
H2.Body.Reader.schedule_read
H2.Body.Reader.close
H2.Body.Writer.write_string
H2.Body.Writer.flush
H2.Body.Writer.close
```

This is stricter than the current client code, but it is the correct server invariant. The client code already documents the important split: connection owns flow reads/writes and shutdown, while the multiplexer is socket-free and adapts `ocaml-h2` callbacks into Eta stream state. 

### 4.3 Reader fiber

The reader fiber owns only socket reads. It does **not** feed `H2.Server_connection` directly.

```ocaml
let reader_loop t =
  let scratch = Bigstringaf.create t.config.read_buffer_size in
  let cs = Cstruct.of_bigarray scratch in
  let rec loop () =
    match Eio.Flow.single_read t.flow cs with
    | 0 -> enqueue t Ingress_eof
    | n ->
        (* Copy or hand off a stable buffer. Do not reuse scratch until owner acks. *)
        let owned = Bigstringaf.create n in
        Bigstringaf.blit scratch ~src_off:0 owned ~dst_off:0 ~len:n;
        enqueue_and_wait t (Ingress { bytes = owned; off = 0; len = n; ack });
        loop ()
    | exception End_of_file ->
        enqueue t Ingress_eof
  in
  loop ()
```

A zero-copy variant can pool buffers later. The MVP can copy at this boundary to preserve ownership and avoid subtle reuse bugs.

### 4.4 Owner loop

The owner loop has three responsibilities:

1. Drive inbound protocol.
2. Drive outbound writer.
3. Coordinate handler/body/response command promises.

```ocaml
let rec owner_loop t =
  match Eio.Stream.take t.commands with
  | Ingress { bytes; off; len; ack } ->
      observe_inbound_security_or_fail t bytes ~off ~len;
      ignore (H2.Server_connection.read t.h2 bytes ~off ~len);
      drain_writes t;
      Eio.Promise.resolve ack ();
      owner_loop t

  | Ingress_eof ->
      ignore (H2.Server_connection.read_eof t.h2);
      drain_writes t;
      close_connection t

  | Request_body_read (stream_id, resolver) ->
      arm_one_request_body_read t stream_id resolver;
      drain_writes t;
      owner_loop t

  | Response_start (stream_id, response) ->
      start_response t stream_id response;
      drain_writes t;
      owner_loop t

  | Response_chunk (stream_id, chunk, resolver) ->
      write_response_chunk_and_flush t stream_id chunk resolver;
      drain_writes t;
      owner_loop t

  | Response_close (stream_id, resolver) ->
      close_response_writer t stream_id;
      drain_writes t;
      Eio.Promise.resolve resolver (Ok ());
      owner_loop t

  | Handler_failed (stream_id, error) ->
      respond_error_or_reset t stream_id error;
      drain_writes t;
      owner_loop t

  | Shutdown policy ->
      begin_shutdown t policy;
      drain_writes t
```

### 4.5 `H2.Server_connection.create` callback

The server callback from `ocaml-h2` should convert `reqd` to an Eta server request and fork the handler, but it must not let the handler mutate `reqd` directly.

```ocaml
let on_request t reqd =
  let request = Request_adapter.of_reqd t reqd in
  let stream_id = request.stream_id in

  t.streams |> Hashtbl.add stream_id {
    reqd;
    request_body = H2.Reqd.request_body reqd;
    response_writer = None;
    handler_cancel = None;
    response_started = false;
    request_done = false;
  };

  let run_handler () =
    let rt = t.runtime_factory ~sw:t.sw ~connection:t.connection_info () in
    let effect =
      Eta_http.Observability.Server.Tracer.request
        ~enabled:t.config.enable_otel
        handler request
    in
    match Eta.Runtime.run rt effect with
    | Eta.Exit.Ok response ->
        enqueue t (Response_start (stream_id, response_to_start response))
    | Eta.Exit.Error cause ->
        let error = Error.of_cause request cause in
        enqueue t (Handler_failed (stream_id, error))
  in

  let cancel = fork_handler_fiber t ~stream_id run_handler in
  record_cancel t stream_id cancel
```

The raw `reqd` stays in the owner’s stream table.

### 4.6 Request body adapter

Server request body should be Eta-native and demand-driven:

```ocaml
let read body =
  Effect.sync (fun () ->
    let promise, resolver = Eio.Promise.create () in
    enqueue body.conn (Request_body_read (body.stream_id, resolver));
    Eio.Promise.await promise)
  |> Effect.bind (function
       | Ok (`Chunk bytes) -> Effect.pure (Some bytes)
       | Ok `Eof -> Effect.pure None
       | Error error -> Effect.fail error)
```

Owner-side read:

```ocaml
let arm_one_request_body_read t stream_id resolver =
  let state = find_stream t stream_id in
  if state.request_done then resolve resolver (Ok `Eof)
  else
    H2.Body.Reader.schedule_read state.request_body
      ~on_read:(fun bs ~off ~len ->
        let chunk = Bytes.create len in
        Bigstringaf.blit_to_bytes bs ~src_off:off chunk ~dst_off:0 ~len;
        resolve resolver (Ok (`Chunk chunk)))
      ~on_eof:(fun () ->
        state.request_done <- true;
        resolve resolver (Ok `Eof))
```

Important: schedule at most one body read per consumer demand. The existing client-side async body adapter documents exactly this backpressure property: each consumer demand arms at most one upstream h2 read and never buffers more than one chunk ahead. 

`discard ?drain` should:

* `drain=false` default: close/reset the stream-side request body and release any blocked reader.
* `drain=true`: read and discard up to configured `Drain_up_to n`, then close/reset if still not EOF.

This avoids unbounded drain on malicious uploads.

### 4.7 Response writer adapter

Handlers return a backend-neutral `Eta_http.Server.Response.t`. The owner converts it to h2 operations.

Fixed response:

```ocaml
match response.body with
| Empty ->
    H2.Reqd.respond_with_string reqd h2_response ""
| Fixed chunks ->
    let body = Bytes.concat Bytes.empty chunks |> Bytes.unsafe_to_string in
    H2.Reqd.respond_with_string reqd h2_response body
| Stream stream ->
    let writer = H2.Reqd.respond_with_streaming reqd h2_response in
    state.response_writer <- Some writer;
    fork_response_pump stream
```

Streaming response pump:

```ocaml
let rec pump () =
  Server.Response.Body.read stream
  |> Eta.Effect.bind (function
       | None ->
           trailers_effect
           |> Eta.Effect.bind (fun trailers ->
                enqueue_effect (Response_trailers (stream_id, trailers)))
           |> Eta.Effect.bind (fun () ->
                enqueue_effect (Response_close stream_id))
       | Some chunk ->
           enqueue_effect_wait (Response_chunk (stream_id, chunk))
           |> Eta.Effect.bind pump)
```

Owner-side write:

```ocaml
let write_response_chunk_and_flush t stream_id chunk resolver =
  let writer = response_writer t stream_id in
  H2.Body.Writer.write_string writer (Bytes.unsafe_to_string chunk);
  H2.Body.Writer.flush writer (fun result ->
    enqueue t (Flush_result (stream_id, result, resolver)))
```

The pump must not read the next application chunk until the flush callback reports progress. Existing client upload code already states this rule for body writers: streaming uploads must await the `ocaml-h2` flush callback before pulling the next chunk, because it is the only transport-progress/backpressure signal. 

### 4.8 Writer draining

For MVP, keep write draining inside the owner loop:

```ocaml
let rec drain_writes t =
  match H2.Server_connection.next_write_operation t.h2 with
  | `Write iovecs ->
      let written =
        Eio.Flow.single_write t.flow (Writer.cstructs_of_iovecs iovecs)
      in
      H2.Server_connection.report_write_result t.h2 (`Ok written);
      drain_writes t
  | `Yield ->
      H2.Server_connection.yield_writer t.h2 (fun () ->
        enqueue t Wake_writer)
  | `Close code ->
      H2.Server_connection.report_write_result t.h2 `Closed;
      close_connection_with_code t code
```

If later split into a writer fiber, the writer fiber may call `Eio.Flow.single_write`, but only the owner may call `report_write_result`.

The existing writer adapter already preserves `Bigstringaf` iovec slice ownership by converting h2 iovecs to `Cstruct.t` views and writing them with Eio. Reuse that conversion logic. 

### 4.9 Cancellation and shutdown

Cancellation behavior should be explicit:

* **Client `RST_STREAM` before handler completes**: owner records stream reset, fails pending body reads/writes, cancels the handler fiber, closes request/response body state, records span error.
* **Handler fails before headers**: owner sends mapped error response.
* **Handler fails after headers**: owner sends `RST_STREAM INTERNAL_ERROR` if stream is still open; otherwise no response.
* **Handler cancelled by Eta timeout**: same as handler failure, but error kind is cancellation/timeout.
* **Connection read EOF**: fail all active streams with `Connection_closed`.
* **Server shutdown graceful**: stop accepting new connections, send GOAWAY on each h2 connection, allow active streams until grace deadline, then close.
* **Server shutdown immediate**: close flows and cancel handler scopes.

The existing h2 admission/stream-state invariant is worth preserving on the server side too: reset streams should remain counted until local cleanup/release, preventing reset churn from evading concurrency limits.

## 5. Multicore strategy

### Default

Use `Eio.Net.run_server` with:

* `Single_domain` if no domain manager is supplied.
* `Recommended` if a domain manager is supplied: `Domain.recommended_domain_count () - 1` additional domains.
* `Additional n` for explicit tuning.

Do not invent a request scheduler for the MVP.

### Ownership model

Each accepted connection belongs to the domain/fiber where Eio runs its handler. All mutable h2 connection state is confined to that connection owner. Handler Eta effects run as child fibers under that connection’s switch, but raw h2 state is accessed only by owner commands.

This is more conservative than sharing `H2.Server_connection.t` behind a mutex. It also fits `ocaml-h2`’s Sans-IO shape and Eta’s existing client lesson that Eta must own the Eio loops, wakeups, admission, error mapping, and public lifecycle around h2. 

### Handler dependencies

Because `run_server` with additional domains may invoke handlers on multiple domains, handler dependencies must be:

* immutable,
* internally synchronized,
* per-domain/per-connection constructed,
* or explicit offload pools designed for cross-domain use.

Do not share domain-bound clients. The existing `Eta_http.Client.t` stores an owner domain and rejects use from another domain, so server examples must avoid closing over one client shared across accept domains. 

### CPU and blocking offload

HTTP/2 connection state must not be offloaded. CPU-bound application work can use `eta_par` explicitly. Blocking work can use `Eta_blocking.run` or a configured blocking pool. Only ordinary application values cross those boundaries; no `reqd`, h2 body reader, h2 body writer, or connection handle may cross.

A good handler pattern:

```ocaml
let handler req =
  Eta_http.Server.Body.read_all req.body
  |> Eta.Effect.bind (fun bytes ->
       Eta_par_effect.run_cpu "parse-json" (fun () -> decode bytes))
  |> Eta.Effect.bind (fun value ->
       Eta_blocking.run ~name:"db.query" (fun () -> query_db value))
  |> Eta.Effect.map response_of_result
```

## 6. OpenTelemetry-by-default design

Add:

```ocaml
Eta_http.Observability.Server.Semconv
Eta_http.Observability.Server.Tracer
Eta_http.Observability.Server.Meter
```

Server tracing should be enabled by default when the runtime has a tracer. It should not link `eta_otel`; it should emit through Eta’s tracer/meter capabilities only, just like the current HTTP client observability layer. The current client tracer uses Eta capabilities and supports suppression to avoid recursive exporter spans; server code can reuse the same principle for internal/self calls. 

### Span lifecycle

Span name:

```text
HTTP GET
HTTP POST
```

Kind:

```ocaml
Eta.Capabilities.Server
```

Start when final request headers are accepted. End when:

* response body closes,
* stream resets,
* handler fails before response,
* connection closes.

Extract W3C context from incoming headers and run the handler under that context.

Attributes:

```text
http.request.method
url.scheme
url.path
url.query.redacted or absent by default
network.protocol.name = "http"
network.protocol.version = "2"
server.address
server.port
client.address
client.port
http.response.status_code
eta_http.server.connection_id
eta_http.server.stream_id
eta_http.error.kind
eta_http.error.layer
error.type
```

Do not emit request body, response body, `authorization`, `cookie`, `set-cookie`, raw query strings, or raw fragments. Existing HTTP semconv helpers redact full URLs by default; server helpers should keep the same default. 

### Metrics

Through `Eta.Effect.metric_update`:

```text
eta_http.server.connections.active        Gauge
eta_http.server.streams.active            Gauge
eta_http.server.requests.total            Counter_monotonic
eta_http.server.requests.in_flight        Gauge
eta_http.server.request.body.bytes        Counter_monotonic
eta_http.server.response.body.bytes       Counter_monotonic
eta_http.server.stream.resets             Counter_monotonic
eta_http.server.protocol.errors           Counter_monotonic
eta_http.server.shutdown.active           Gauge
```

Since Eta’s meter surface currently supports counters and gauges, avoid designing around histograms until the meter grows them.

## 7. Error model and typed failure mapping

Do not expand the existing client-oriented `Eta_http.Error.kind` for the server MVP. Add `Eta_http.Server.Error`. It can project into `Eta_http.Error.t` for shared logging/projection, but server code needs server-specific context: stream id, target, handler layer, response write state.

Mapping before response headers:

| Error                                                |             HTTP response |
| ---------------------------------------------------- | ------------------------: |
| `Bad_request`, invalid pseudo-header, invalid header |                       400 |
| request body too large                               |                       413 |
| request timeout                                      |                       408 |
| stream admission rejected / overloaded               |                       503 |
| handler typed failure not otherwise mapped           |                       500 |
| protocol error requiring connection close            | no response, GOAWAY/close |
| client reset / connection closed                     |               no response |

Mapping after response headers:

| Error                         | Action                                               |
| ----------------------------- | ---------------------------------------------------- |
| handler fails while streaming | `RST_STREAM INTERNAL_ERROR`                          |
| response body read fails      | `RST_STREAM INTERNAL_ERROR`                          |
| trailers invalid              | `RST_STREAM INTERNAL_ERROR`                          |
| socket write failure          | close connection                                     |
| cancellation due shutdown     | graceful: finish if possible; immediate: reset/close |

Application errors should be explicit:

```ocaml
let handler =
  Eta_http.Server.Handler.map_error
    (function
      | `Not_found path ->
          Eta_http.Server.Error.make ~method_:"GET" ~target:path
            (Bad_request { message = "not found" })
      | `Db_down ->
          Eta_http.Server.Error.make ~method_:"*" ~target:"*"
            (Handler_failed { message = "database unavailable" }))
    app_handler
```

Exceptions raised by `Eta.Effect.sync` are defects, not typed failures, according to Eta’s effect contract; server adapter code should catch `Eta.Exit.Error` causes at the runtime boundary and map defects to `Handler_failed`/500 with safe diagnostics.

## 8. Test plan

### Shared tests: `test/http_common/server_common_suites.ml`

Add backend-neutral tests for:

* `Server.Request` construction from pseudo-headers.
* header validation and redaction.
* `Server.Response` helpers.
* error-to-status mapping.
* semconv attributes and redacted target/query.
* trace context extraction.
* default error renderer.

### Eio h2c tests: `test/http_eio/test_eta_http_eio_h2_server.ml`

Core behavior:

* h2c healthz GET.
* POST echo with streaming request body.
* 100 concurrent streams on one connection.
* response trailers.
* streaming response with many chunks.
* handler returns 404/500.
* early response without fully reading request body.
* invalid request headers -> 400 or stream error.

The current tests already prove similar client-side h2 invariants: 100 concurrent GETs, upload flow-control resume, reset/admission release, cancel release, trailers, and body discard release. Port those patterns to the server path.  

Backpressure:

* handler reads request body one chunk at a time; client upload stalls until reads resume.
* response stream pump does not pull next chunk until h2 flush callback.
* socket slow writer does not unbounded-buffer response chunks.
* request body discard resets by default and bounded-drains only with configured drain policy.

Cancellation:

* client `RST_STREAM` cancels handler.
* handler cancellation resets stream and releases request body.
* connection EOF fails all pending request body reads.
* graceful shutdown sends GOAWAY and waits for in-flight streams.
* immediate shutdown closes flow and cancels handlers.
* switch cancellation is classified as connection closed, not protocol violation. There is already a red-test style comment in the current h2 client tests around cancellation classification; the server should avoid repeating that bug. 

Security/adversarial:

* rapid reset churn.
* CONTINUATION flood.
* HPACK/header block cap.
* SETTINGS flood.
* PING flood.
* empty DATA frame flood.
* WINDOW_UPDATE overflow.
* slow request body.
* malformed pseudo-headers.
* oversized header names/values.

The current client security envelope scans raw inbound h2 bytes before feeding `ocaml-h2`; the server needs the analogous client-to-server scanner or a generalized bidirectional security observer. 

Observability:

* span kind is server.
* traceparent becomes parent context.
* status code annotation on success.
* error annotation on typed failure.
* request/response bytes metrics.
* active connection/stream gauges.
* URL query redacted by default.
* no raw sensitive headers.

Multicore:

* run h2c server with `Additional 1` or `Recommended` in tests.
* many clients hit the server concurrently.
* per-connection owner invariant holds.
* handler dependencies are thread-safe or per-domain.
* a deliberately shared domain-bound client produces a documented failure or is avoided by examples.

Interop:

Extend `http-testsuite` to add Eta as a server target:

```text
curl/nghttp -> eta_http_eio h2c server
eta_http client -> eta_http_eio h2c server
```

The testsuite already has real-server interop/differential testing structure with nginx/Caddy, curl normalization, and adversarial fixtures. Add Eta server as another target rather than inventing a second harness.

### Phase 5: cancellation, shutdown, and typed errors

Implement:

* graceful GOAWAY.

## 10. Risks and non-goals

### Risks

* `ocaml-h2` server callbacks may synchronously invoke body/response callbacks in places where owner-loop reentrancy matters. Keep callback bodies tiny: record state, enqueue, return.
* Server-side security scanning needs either a bidirectional generalization of `Eta_http.H2.Security` or a new inbound scanner. Do not feed hostile client frames straight into `H2.Server_connection.read`.
* Handler code under multicore can accidentally close over non-thread-safe state. Document this in `Eta_http_eio.Server.run_h2c`.
* Response streaming can accidentally pull faster than h2 flow control. The flush-ack rule must be tested.
* Graceful shutdown semantics depend on what `ocaml-h2` exposes for server GOAWAY. If last-stream control is limited, prefer conservative GOAWAY/close semantics and document them.

### Non-goals

* No HTTP/1.1 server in the MVP.
* No h2c upgrade from HTTP/1.1 in the MVP; use prior-knowledge h2c.
* No Java/Spring-style routing framework.
* No middleware stack as a core abstraction.
* No static-file server.
* No cookie/session layer.
* No server push.
* No cross-domain HTTP/2 request work stealing.
* No raw request/response body telemetry.
* No selective retry/replay around GOAWAY.

### Boundary changes before coding

Two small boundary moves are worth doing first:

1. Add `Eta_http.Server` as a shared API before any Eio loop. This prevents `eta_http_eio` from inventing a backend-specific request shape.
2. Add `Eta_http.Observability.Server` rather than overloading the existing client semconv helpers. The current observability modules are explicitly client-oriented, including client request span wrappers and client connection metrics. 

The blast radius should be small: new modules, new exports, new tests. Existing client code should not need churn.

Overall: implement h2c server support by mirroring the strongest parts of the current h2 client adapter—Sans-IO ownership, typed errors, body-stream release semantics, security envelope, and Eta observability—while tightening the ownership rule so raw HTTP/2 state remains single-owner per connection.
