# Section A: Current known blocker and root-cause plan

## A1. What is most likely happening in `tls.13 HTTPS server H2 ALPN request`

The observed sequence is important: ALPN negotiates `h2`, the client writes the H2 connection preface/settings and request HEADERS, the server TLS wrapper receives enough encrypted bytes for both records, but one run showed `SSL_read` returning only the first 58 plaintext bytes. That is exactly the size class of “preface plus SETTINGS-ish” data, not the full request. The H2 server reader then enqueues that first ingress; the H2 owner loop can be seen waiting for commands in some interleavings. The dirty H2 server code’s reader creates an ingress `ack` promise but does not await it; the owner loop drains writes, takes commands, handles ingress, schedules idle timeouts, and loops.

The most likely root cause is therefore **the TLS memory-BIO wrapper’s read-driving contract**, not H2 frame serialization itself. `tls_eio.ml` drives OpenSSL with separate `feed_bio` and `drain_bio` helpers, split flow read/write locks, and an `ssl_mutex`; `single_read` calls `SSL_read`, drains writes, and on WANT_READ performs a blocking underlying flow read into the BIO.  The missing pieces are: reliable handling of `SSL_pending`, reliable draining of already-decrypted plaintext before blocking on the underlying flow, and a full-duplex rule that prevents the write side from consuming or blocking on inbound encrypted bytes.

### Primary suspected bug

`Flow_impl.single_read` can return one plaintext slice from OpenSSL, then on the next call block in `feed_bio` even though OpenSSL may already have decrypted plaintext or buffered TLS record bytes available. In a H2 server, that means ocaml-h2 sees the client preface/settings but not the HEADERS until some later scheduling event or timeout wakes the system. The fact that a widened timeout sometimes passes after roughly 3.5–3.7 seconds points to progress eventually occurring, not to malformed bytes.

### Plausible alternate cause 1: writer-side `SSL_write WANT_READ` consumes the read path

`Flow_impl.single_write` handles `SSL_write WANT_READ` by calling `feed_bio`. That means a writer fiber can perform an underlying socket read while the reader fiber is the logical owner of application ingress. With separate `read_mutex`/`write_mutex`, the split is directionally better than one coarse flow lock, but **both read and write SSL paths can still cause network reads**. In a TLS-over-H2 server, where ocaml-h2 may immediately emit SETTINGS/ACK bytes while the client sends HEADERS, this can create unfairness or serialization around `ssl_mutex` and `read_mutex`.

### Plausible alternate cause 2: H2 owner/reader command handoff lost deterministic backpressure

The H2 server `reader_loop` enqueues `Ingress { bytes; off; len; ack }` and immediately loops without awaiting `ack`; the owner loop eventually resolves the ack in `handle_command`.  That is not automatically wrong, but it removes a deterministic “owner has consumed this ingress” handoff. If TLS returns the first plaintext fragment and then the reader blocks inside TLS instead of immediately returning the second plaintext fragment, the owner can run out of commands and sleep on `Eio.Stream.take`.

### Plausible alternate cause 3: dirty H2 writer coalescing changed timing, not correctness

`lib/http_eio/h2/writer.ml` now coalesces ocaml-h2 iovecs into one Cstruct before `Eio.Flow.write`. This likely reduces TLS record fragmentation and makes the stall less frequent or different, but it does not fix the underlying contract: HTTP/2 over TLS must work when frames, frame headers, HPACK blocks, and TLS records are fragmented arbitrarily.

## A3. Concrete fix plan

### Step 1: fix verified TLS Cstruct offset bug

In `lib/http_eio/tls/tls_eio.ml`, both `single_read` and `single_write` destructure the `Cstruct.t` to get `off` but then pass offset `0` into `Openssl.read`/`Openssl.write`. The OpenSSL stubs honor the provided bigarray offset, so passing `0` is wrong for any `Cstruct.sub` or nonzero-offset view.

Fix shape:

```ocaml
let { Cstruct.buffer; off = cstruct_off; len } = Cstruct.to_bigarray buf in
Openssl.read t.ssl storage cstruct_off len
```

For writes, recurse over the logical Cstruct slice:

```ocaml
let rec write_buf logical_off remaining =
  let written =
    with_ssl t (fun () ->
      Openssl.write t.ssl storage (cstruct_off + logical_off) remaining)
  in
  ...
```

Tests:

```ocaml
test_tls_eio_single_read_respects_cstruct_offset
test_tls_eio_single_write_respects_cstruct_offset
```

These should create a TLS client/server pair, read into `Cstruct.sub dst 5 n`, assert guard bytes remain unchanged, and write from `Cstruct.sub src 7 n`, assert the peer receives exactly the subrange.

### Step 2: make TLS read delivery pending-aware

In `tls_eio.ml`:

1. Add a fast path before blocking on raw flow:

   * if `Openssl.ssl_pending t.ssl > 0`, call `SSL_read` again without `feed_bio`;
   * after every successful `feed_bio`, call `SSL_read` until either it returns application bytes or a real WANT_READ with no pending bytes.
2. Drain wbio after every SSL operation that may produce TLS records.
3. Never hold `ssl_mutex` while blocking in `Flow.single_read` or `Flow.write`.
4. Instrument a test-only counter: raw reads, SSL reads, `ssl_pending` observations, WANT_READ/WANT_WRITE transitions.

Test:

```ocaml
test_tls_eio_h2_server_drains_second_plaintext_record_without_external_wakeup
```

Trigger: client sends two TLS records back-to-back: record 1 contains H2 preface/settings, record 2 contains HEADERS. The server TLS `single_read` may return only record 1 first, but the second call must return record 2 immediately without waiting seconds or requiring unrelated wakeups.

### Step 3: stop writer-side raw reads from starving ingress

Change `single_write` WANT_READ handling. A TLS writer should drain wbio and yield/retry, or request that the read pump feed inbound bytes, but it should not independently block on the underlying flow unless the design has a single centralized TLS pump. The cleaner design is:

* one TLS connection owner/pump owns raw flow reads into rbio and raw writes from wbio;
* application `single_read`/`single_write` communicate with that pump via queues/conditions;
* this pump is the only code that blocks on the raw flow.

This is broader but substantially safer for public edge use.

### Step 4: make H2 ingress handoff deterministic

In `lib/http_eio/h2_server_connection.ml`:

* Either remove the unused ingress `ack` field or make it meaningful.
* Prefer: reader enqueues bounded ingress and then yields until owner consumes when the command queue is above a low watermark. Do not let a malicious client fill command queues with ingress while response/body fibers are also trying to enqueue commands.
* Add a test where the TLS/read path returns `58` bytes, then the remaining HEADERS, and assert the owner loop produces response HEADERS without sleeps.

### Step 5: prove the blocker fix

Acceptance:

```sh
EIO_BACKEND=posix dune exec test/http/run.exe -- -q "tls:HTTPS server H2 ALPN request"
EIO_BACKEND=posix dune runtest test/http --force
nix --option eval-cache false develop -c dune build @red-probes
nix --option eval-cache false develop -c dune exec http-testsuite/test/red_probes/tls_frag/run.exe
```

The current checked-in red-probe summary claims broad fixes and verification, including `dune build @red-probes` and `dune runtest test/http --force`, but the prompt says the final current no-debug result after cleanup was not completed; I would treat those checked-in claims as useful history, not release evidence.

# Section C: Unknown bug hunt

## C1. P0: unsafe default H2 concurrency and stream table sizing

`server_types.ml` defaults H2 `max_concurrent_streams` to `Int32.max_int`. In `h2_server_connection.run`, the per-connection `streams` hashtable is created from that configured max. Even if OCaml’s hashtable does not allocate literally 2.1B buckets, the intent is still unsafe: it advertises effectively unbounded H2 concurrency and removes the most important per-connection DoS guard.

**Minimal trigger:** start default H2C server and open streams until resource pressure or admission failure. Expected edge behavior: bounded SETTINGS `MAX_CONCURRENT_STREAMS`, bounded server state, deterministic refusal or stream reset.

**Fix sketch:**

* Default `max_concurrent_streams` to 100 or 128.
* Add `server.limits.max_active_streams_per_connection` independent of ocaml-h2 config.
* Allocate tables with `min configured 256` or a fixed small initial capacity.
* Add tests:

  * `test_server_config_default_h2_max_concurrent_streams_is_bounded`
  * `test_h2c_server_default_refuses_streams_over_limit`
  * `test_h2_server_stream_table_initial_capacity_is_capped`

## C2. P1: TLS `Cstruct.off` ignored

The TLS wrapper passes offset `0` into OpenSSL reads/writes even when the caller supplied a sub-Cstruct. This is a classic corruption bug: bytes may be written into the wrong slice or written from the wrong slice.

**Minimal trigger:** read TLS plaintext into a Cstruct subview with sentinel bytes before and after. If sentinels change or the returned plaintext lands at offset 0, the bug is confirmed.

**Fix sketch:** use the Cstruct view’s actual bigarray offset in both `single_read` and `single_write`.

## C3. P1: H2 server ingress filter buffers incomplete large frames before cap

`filter_ingress` concatenates `filter_pending` and current bytes into a string and waits until a complete frame is present before forwarding or dropping. The normal `append_ingress` cap applies after filtering. A peer can send a large frame header, then drip or withhold the payload, forcing `filter_pending` growth and repeated string concatenation before ocaml-h2 sees anything.

**Minimal trigger:** H2C preface + SETTINGS + DATA/unknown frame header with length `0xffffff`, then drip 1 byte every tick. Expected: immediate connection error if frame size exceeds local max, or bounded buffering and timeout.

**Fix sketch:**

* Replace string-concat frame filtering with a state machine:

  * parse 9-byte header;
  * validate `length <= local_max_frame_size`;
  * stream frame payload through without waiting for whole frame unless it is a small frame whose contents must be inspected;
  * for DATA-after-RST dropping, skip payload bytes by counter, not by buffering.
* Apply cap before `filter_pending` growth.
* Add `h2_frames.huge_incomplete_frame_is_bounded`.

## C4. P1: TLS write side can consume inbound bytes

`SSL_write` WANT_READ currently leads to `feed_bio`, which performs raw flow reads. That is risky in an effect/fiber runtime because the application’s logical write path now owns read progress and can block behind read locks or consume bytes needed to wake the H2 reader path.

**Minimal trigger:** TLS pair where server is writing H2 SETTINGS/response while client sends request HEADERS; force OpenSSL WANT_READ during write by post-handshake behavior or simulated OpenSSL wrapper. Assert the reader still receives inbound plaintext promptly.

**Fix sketch:** centralize TLS I/O in a pump. Short-term, on writer WANT_READ: drain wbio, yield, and retry only after read-side notification; do not block on raw `single_read`.

## C5. P1: H2 EOF/incomplete header policy is too implicit

`Ingress_eof` currently calls `read_eof`, fails open request bodies, shuts down the H2 connection, drains writes, marks closed, and closes the flow. That is terminally safe but protocol-blunt. It does not give precise behavior for incomplete HEADERS, incomplete DATA bodies, or graceful half-close.

**Minimal trigger:** client sends HEADERS without END_HEADERS, then half-closes. Current red finding leaves `headers_without_end_headers` as policy gap. 

**Fix sketch:**

* Track open header block and active body state at Eta layer.
* On EOF with incomplete header block: GOAWAY `PROTOCOL_ERROR` then close.
* On EOF with incomplete request body: RST_STREAM `CANCEL` or `PROTOCOL_ERROR`, fail handler body reads, then GOAWAY if connection cannot continue.
* Add deterministic tests with no sleeps.

## C6. P2: H1 client response TE empty-token acceptance

`validate_transfer_encoding` for H1 client responses filters empty tokens before deciding whether TE is valid. That makes `Transfer-Encoding: chunked,` equivalent to `chunked`.  For request parsing, Eta is stricter and rejects the same shape. 

**Minimal trigger:** malicious H1 server sends:

```http
HTTP/1.1 200 OK
Transfer-Encoding: chunked,

0
```

Expected: protocol violation, close, no body reuse.

**Fix sketch:** preserve empty tokens and reject them. If compatibility is desired, put it behind an explicit non-edge compatibility mode.

## C7. P2: H1 bare CR request line timeout

Checked-in H1 smuggle findings keep `bare_cr_request_line` as the only remaining non-PASS case: Eta waits for request-header timeout instead of returning immediate 400. 

**Minimal trigger:** send `GET / HTTP/1.1\rHost: x\r\n\r\n`.

**Fix sketch:** while scanning for CRLF in request head, detect a CR not followed by LF before timeout and return `400 Bad Request; Connection: close`.

## C8. P2: H2 security “rate” errors are not real rates

Security errors include names like `Ping_rate_exceeded` and `Settings_churn_rate_exceeded`, but the scanner’s documented behavior is cheap frame-envelope counting over a connection, not a time-windowed rate limiter. 

**Minimal trigger:** send 11 SETTINGS over an hour vs 11 SETTINGS in a millisecond; current counter semantics likely treat both the same.

**Fix sketch:** either rename to `*_count_exceeded` or implement token-bucket/time-window rate limiting using injected time. For edge use, counts are acceptable but names must not mislead operators.

## C9. P2: H2 benchmark confidence gap

The server-load harness maps H2 benchmark concurrency to one TCP connection and protocol-level parallelism. That is a valid H2 multiplexing test, especially for flow control and stream scheduling, but it does not measure public-edge realities: accept loop, TLS handshakes, per-connection memory, file descriptors, and many-client fairness.

**Fix sketch:** add matrix dimensions:

```text
h2_connections = [1; 4; 16; 64]
h2_streams_per_connection = [1; 16; 64; 128]
```

Keep current one-connection test as “single-connection multiplex stress,” not the headline H2 edge benchmark.

## Bottom line

The core validation work looks much better than the prior review suggests: many H2 frame envelope checks, H2 request/response validators, H1 request framing fixes, redaction fixes, method case-sensitivity fixes, and H1 response body suppression fixes appear present and covered. The checked-in red findings also indicate broad progress.

The release decision should still be **no for public edge today** because H2/TLS is not deterministic, TLS Cstruct offset handling is wrong, H2 default concurrency is unsafe, and a few remaining policy gaps sit exactly where edge attackers operate: fragmentation, incomplete frames, GOAWAY/RST lifecycle, and slow/stalled peers.
