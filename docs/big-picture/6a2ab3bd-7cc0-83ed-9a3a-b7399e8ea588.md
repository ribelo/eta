# Eta HTTP server production design and gap-closure plan

This plan is grounded in the uploaded source package and keeps the current boundary: **`eta_http` owns backend-neutral HTTP types and pure protocol helpers; `eta_http_eio` owns Eio sockets, fibers, TLS flow wrapping, listener lifecycle, and protocol connection ownership.** 

## 0. Current-state summary

The repository already has a good server substrate, but it is not yet a production HTTP server. The strongest implemented server path is **h2c** in `lib/http_eio/h2_server_connection.ml` and `lib/http_eio/server.ml`: it has an Eio-owned connection loop, a bounded command queue, request-body commands, response streaming commands, graceful shutdown plumbing, per-connection stats, and h2c tests for fixed responses, echo bodies, unread-body drain, trailers, graceful shutdown, and typed read/write failures.   

The backend-neutral server API exists in `lib/http/server*.ml`: `Server.Request.t`, `Server.Response.t`, `Server.Body.t`, typed `Server.Error.t`, handler helpers, server semantic-convention helpers, tracer wrapper, and server meter helpers. It is usable, but still h2/h2c-shaped: `Server_error.protocol` currently has `H2 | H2c | Unknown`, with no `H1`; `Server.Response.Body.Stream` lacks an explicit optional length; and request trailers are not modeled.   

Major production gaps are **HTTP/1.1 server**, **HTTPS/TLS server**, **ALPN server dispatch**, complete **server limits/timeouts**, and a server-facing interop/adversarial suite. Current TLS code is client-side only: `Tls_eio.client_of_flow` wraps an outgoing flow, performs the client handshake, verifies peer certificates, and exposes selected ALPN; the low-level OpenSSL binding is described as a client context with peer verification.

# 1. Target architecture

## 1.1 Package layout

### `eta_http`: backend-neutral contract and protocol helpers

Keep `eta_http` free of Eio. It should own:

| Area                    | Target modules                                                                                 | Ownership                                                                                                                                                                       |
| ----------------------- | ---------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Public server model     | `Server`, `Server.Request`, `Server.Response`, `Server.Body`, `Server.Error`, `Server.Handler` | Handler-facing API, errors, body contracts, response construction.                                                                                                              |
| Server config shape     | `Server.Config`, `Server.Limits`, `Server.Timeouts`                                            | Backend-neutral semantics: header/body limits, trailers policy, error-response policy, observability flags.                                                                     |
| H1 protocol helpers     | `H1.Request_parse`, `H1.Response_write`, `Body.Chunked`                                        | Pure request parsing and response serialization decisions. Current `H1.Parse`/`H1.Write` are response/client oriented and should be split rather than overloaded.               |
| H2 protocol helpers     | `H2.Admission`, `H2.Security`, `H2.Stream_state`, `H2.Frame`                                   | Pure H2 lifecycle/security accounting; no socket ownership. Existing H2 helpers already follow this pattern.                                                                    |
| TLS policy descriptions | `Tls.Config.Client`, `Tls.Config.Server`, `Tls.OpenSSL`                                        | Policy and low-level state-machine bindings, not Eio flow ownership.                                                                                                            |
| ALPN                    | `Transport.Alpn`, `Transport.Dispatch`                                                         | Pure protocol decision logic. The existing ALPN state machine and `dispatch_alpn` are reusable, but server dispatch should not require a client `Request.t` for error context.  |

### `eta_http_eio`: Eio transport and lifecycle

Refactor `eta_http_eio` into explicit server transport modules:

| Target module          | Responsibility                                                                                                   |
| ---------------------- | ---------------------------------------------------------------------------------------------------------------- |
| `Server`               | Public start/run/shutdown/stats API for h1, h2c, HTTPS, and ALPN.                                                |
| `Server_listener`      | Accept loop, domain policy, connection registry, shutdown stop promise.                                          |
| `H1_server_connection` | Own one HTTP/1.x connection flow; parse requests; run handler; serialize responses; keep-alive policy.           |
| `H2_server_connection` | Own one HTTP/2 connection flow; reuse/refine current h2c implementation; support both h2c and TLS+h2.            |
| `Tls_server`           | Wrap accepted TCP flow into TLS server flow; handshake timeout; ALPN selection; server epoch metadata.           |
| `Alpn_server`          | Route post-handshake TLS flow to H1 or H2 connection owner.                                                      |
| `Server_stats`         | Thread-safe counters and snapshots for connections, requests, streams, bytes, timeouts, resets, protocol errors. |
| `Server_runtime`       | Default and custom `runtime_factory`; per-connection/per-request runtime construction policy.                    |

The current `eta_http_eio` package already depends on `eta_http`, `eta_eio`, `eta_stream`, `eio`, `h2`, `hpack`, `faraday`, `bigstringaf`, `cstruct`, `domain-name`, and `ipaddr`, so the target design does not require a new package boundary. 

## 1.2 Connection ownership invariants

Use these invariants across h1, h2c, and HTTPS:

| Invariant                               | Design rule                                                                                                                                                                                                                                               |
| --------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| One flow owner                          | Exactly one protocol connection module owns reads/writes/closes for a flow. Handler fibers never touch `Eio.Flow` or `H2.Server_connection` directly.                                                                                                     |
| One protocol state owner                | For H1, the connection owner serializes parser/writer state. For H2, the existing owner loop remains the only code that calls `H2.Server_connection.read`, `next_write_operation`, `Body.Writer.*`, and `Reqd.*`.                                         |
| Command boundary                        | Handler fibers communicate with the owner through typed commands: request body read/discard, response start/chunk/trailers/close/fail, shutdown. This is already visible in `H2_server_connection.command` and should become the general server pattern.  |
| Pull request bodies                     | Request body reads are demand-driven. The server reads body bytes only when the handler asks, or when configured unread-body drain is required.                                                                                                           |
| Backpressured response bodies           | Streaming responses wait for protocol/flow write completion before pulling the next chunk. H2 already flushes response chunks through the owner; H1 should do the same with flow writes.                                                                  |
| Release once                            | Request and response body release functions run at most once, even on cancellation/error. `Server_body` already enforces single active operation and release-on-EOF semantics; keep that contract.                                                        |
| Graceful shutdown stops admission first | Listener stops accepting; H2 sends GOAWAY; H1 responses include `Connection: close`; active streams/requests complete until deadline; immediate shutdown closes all flows.                                                                                |
| Typed errors only                       | Non-cancellation host exceptions are mapped into `Server.Error.t`. Raw exceptions must not escape owner loops, callbacks, or handler boundaries.                                                                                                          |

## 1.3 Data flow

### Plain H1

```
Eio.Net.run_server
  -> H1_server_connection.run
  -> read request head
  -> build Eta_http.Server.Request.t
  -> run Eta_http.Observability.Server.Tracer.request handler request
  -> encode Eta_http.Server.Response.t
  -> write response
  -> drain/reset unread request body according policy
  -> keep alive or close
```

### Plain h2c

```
Eio.Net.run_server
  -> H2_server_connection.run ~tls:false ~alpn:"h2c"
  -> reader fiber reads bytes and enqueues Ingress
  -> owner loop feeds H2.Server_connection and drains writes
  -> h2 request callback builds Server.Request.t
  -> handler fiber runs runtime
  -> handler commands response start/chunks/trailers/close
  -> owner serializes H2 writes
```

This is a direct refinement of the current h2c implementation, which already has `Ingress`, `Request_body_read`, `Request_body_discard`, `Response_start`, `Response_chunk`, `Response_trailers`, `Response_close`, `Response_failed`, and `Shutdown` commands. 

### HTTPS with ALPN

```
accepted TCP flow
  -> Tls_server.server_of_flow
  -> TLS handshake + ALPN select ["h2"; "http/1.1"]
  -> epoch { tls=true; alpn_protocol }
  -> Alpn_server.dispatch:
       "h2"       -> H2_server_connection.run ~tls:true ~protocol:H2
       "http/1.1" -> H1_server_connection.run ~tls:true
       None       -> H1_server_connection.run ~tls:true
       other      -> close + typed/logged ALPN error
```

# 2. Public API design

## 2.1 `eta_http` server-facing API changes

API-breaking changes are worthwhile here because H1 and TLS need a clearer long-term model.

### Protocol and transport context

```ocaml
module Eta_http.Server : sig
  type protocol = H1_0 | H1_1 | H2 | H2c
  type transport = Plain | Tls of { alpn_protocol : string option }

  module Request : sig
    type peer = { address : string option; port : int option }

    type t = {
      id : string;
      version : Core.Version.t;
      scheme : string;
      authority : string option;
      method_ : string;
      target : string;
      path : string;
      query : string option;
      headers : Core.Header.t;
      body : Body.t;
      trailers : unit -> (Core.Header.t, Error.t) Eta.Effect.t;
      peer : peer;
      tls : bool;
      alpn_protocol : string option;
      stream_id : int option;
      connection_id : string;
    }
  end
end
```

Recommended change: add `connection_id` and request trailers. Keep `tls` and `alpn_protocol` because they are already in `Server.Request.t` and are useful for logs, routing, and auth decisions. 

### Response bodies with explicit stream length

Current `Server.Response.Body.Stream` has only `read` and `release`; H1 needs to choose between `Content-Length`, chunked transfer coding, or close-delimited response without buffering. Add optional length:

```ocaml
module Eta_http.Server.Response.Body : sig
  type stream = {
    length : int option;
    read : unit -> (bytes option, Server_error.t) Eta.Effect.t;
    release : unit -> (unit, Server_error.t) Eta.Effect.t;
  }

  type t =
    | Empty
    | Fixed of bytes list
    | Stream of stream

  val stream :
    ?length:int ->
    ?release:(unit -> (unit, Server_error.t) Eta.Effect.t) ->
    (unit -> (bytes option, Server_error.t) Eta.Effect.t) ->
    t
end
```

Default H1 behavior: fixed bodies get `Content-Length`; stream with `length=Some n` gets `Content-Length`; stream with `length=None` gets chunked for HTTP/1.1 and no trailers unless chunked is active. The existing fixed/stream/trailer response model is a good base. 

### Server errors

Extend the protocol enum and status mapping:

```ocaml
module Eta_http.Server.Error : sig
  type protocol = H1 | H2 | H2c | Unknown

  type kind +=
    | Request_header_too_large of { limit : int; actual : int }
    | Request_headers_too_many of { limit : int; actual : int }
    | Request_target_too_long of { limit : int; actual : int }
    | Unsupported_transfer_encoding of { value : string }
    | Tls_handshake_failed of { message : string }
    | Alpn_rejected of { protocol : string option }
end
```

Current errors cover many server cases, including bad request, invalid header, request body too large, timeout, admission rejection, stream reset, connection closed, protocol error, handler failure, and response write failure. Add H1/TLS-specific variants rather than overloading `Protocol_error`. 

### Backend-neutral config

Put semantic limits in `eta_http` and transport knobs in `eta_http_eio`.

```ocaml
module Eta_http.Server.Config : sig
```

## 2.2 `eta_http_eio.Server` entry points

Keep the current explicit h2c API for compatibility, but add explicit H1 and TLS entry points. The existing `start_h2c`, `run_h2c`, `start_h2c_on_socket`, `run_h2c_on_socket`, `shutdown`, and `stats` form the model. 

```ocaml
module Eta_http_eio.Server : sig
  type t

    | Graceful of Eta.Duration.t

  type protocol =
    | H1

    type protocol_stats = {
      active_connections : int;

      reset_streams : int;
      protocol_errors : int;
      request_bytes : int64;
      response_bytes : int64;
    }

    type t = {
      active_connections : int;
      opened_connections : int;
      closed_connections : int;
      h1 : protocol_stats;
      h2 : protocol_stats;
      tls_handshakes : int;
      tls_handshake_failures : int;
      alpn_h1 : int;
      alpn_h2 : int;
      shutdown_active : bool;
    }
  end

  val start_h1 :
    sw:Eio.Switch.t ->
    net:_ Eio.Net.t ->
    clock:[> float Eio.Time.clock_ty] Eio.Std.r ->
    ?domain_manager:_ Eio.Domain_manager.t ->
    ?config:Config.t ->

  val start_h1_on_socket : ... -> t
  val run_h1_on_socket : ... -> unit

  val run_h2c : ... -> unit
  val start_h2c_on_socket : ... -> t
  val run_h2c_on_socket : ... -> unit

  val start_https :
    sw:Eio.Switch.t ->
    net:_ Eio.Net.t ->
    clock:[> float Eio.Time.clock_ty] Eio.Std.r ->
    tls:tls_config ->
    ?domain_manager:_ Eio.Domain_manager.t ->
    ?config:Config.t ->
    addr:Eio.Net.Sockaddr.stream ->
    Eta_http.Server.handler ->
    t

  val run_https : ... -> unit
  val start_https_on_socket : ... -> t
  val run_https_on_socket : ... -> unit

  val stats : t -> Stats.t

```

### Recommendation

Make `start_https` the production default. Keep `start_h1` and `start_h2c` explicit because h2c is useful for tests, internal service meshes, and cleartext benchmarks, but production Internet-facing usage should be `start_https`.

# 3. Protocol design

## 3.1 HTTP/1.1 server behavior

### Request parsing

Implement `Eta_http.H1.Request_parse` as a pure, bounded parser. It should parse:

| Item                   | Behavior                                                                                                                                                                                                                                          |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Request line           | `METHOD SP request-target SP HTTP/version CRLF`; reject invalid versions and request-target overflow.                                                                                                                                             |
| Header section         | Validate header names/values with existing `Core.Header` rules; reject obs-fold, NUL, CR/LF injection, too many headers, and oversized header section. Current header validation already rejects invalid names and control characters in values.  |
| Body framing           | Support no body, `Content-Length`, and `Transfer-Encoding: chunked`. Reject conflicting `Content-Length`, invalid `Content-Length`, `Transfer-Encoding` other than final `chunked`, and CL+TE request smuggling inputs.                           |
| `Expect: 100-continue` | Configurable: auto-send `100 Continue` when handler first reads the body, reject early if headers/body limit already fails.                                                                                                                       |
| Connection policy      | HTTP/1.1 keep-alive by default unless `Connection: close`; HTTP/1.0 close by default unless `Connection: keep-alive`.                                                                                                                             |
| Pipelining             | Default `Disabled`: process one request at a time and do not pre-read beyond the current body. Optional `Sequential { max_pending }` later.                                                                                                       |

### Response serialization

Implement `Eta_http.H1.Response_write` as pure serialization decisions plus Eio write in `H1_server_connection`.

| Response shape               | HTTP/1.1 wire behavior                                                                                                                                     |
| ---------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `HEAD`, `1xx`, `204`, `304`  | No response body; ignore body chunks; `Content-Length` only when semantically allowed and known.                                                           |
| `Fixed chunks`               | Add `Content-Length` if caller did not provide one; write bytes exactly.                                                                                   |
| `Stream { length = Some n }` | Add `Content-Length: n`; write exactly `n` bytes; protocol error if stream length mismatches.                                                              |
| `Stream { length = None }`   | Use `Transfer-Encoding: chunked` for HTTP/1.1; close-delimited only for HTTP/1.0 or explicit configured fallback.                                          |
| Trailers                     | Only with chunked HTTP/1.1 or H2. Validate trailer names and forbid `Content-Length`, `Transfer-Encoding`, `Connection`, `Host`, and other framing fields. |
| Header normalization         | Do not forward hop-by-hop response headers from application. Either reject them at construction or filter with a logged typed error.                       |

### H1 connection state machine

Recommended state machine:

```
Accept
  -> Read_head
  -> Maybe_read_body_on_demand
  -> Run_handler
  -> Write_response_head
  -> Pump_response_body
  -> Write_trailers
  -> Drain_or_close_unread_request_body
  -> Keep_alive_wait | Close
```

The H1 connection owner should maintain a bounded read buffer and never allow an unbounded parser buffer. `Server.Body.read` should pull from the connection owner through a command queue just like h2c, not read the socket from handler fibers.

## 3.2 HTTP/2 server behavior

Retain the current h2c owner-loop architecture and generalize it:

| Area                | Final behavior                                                                                                                                                                                                                                            |
| ------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| h2c and h2-over-TLS | `H2_server_connection.run` accepts `connection_info` including `tls` and `alpn_protocol`; `run_h2c` becomes a wrapper.                                                                                                                                    |
| Owner loop          | Keep a single owner loop around `H2.Server_connection`; reader fiber enqueues `Ingress` and waits for ack; handler fibers enqueue response/body commands. This matches current design.                                                                    |
| Stream identity     | Populate `Server.Request.stream_id` with the actual H2 stream id where `ocaml-h2` exposes it; otherwise store ordinal in a separate internal field and keep public `stream_id=None`.                                                                      |
| Concurrent streams  | Set `H2.Config.max_concurrent_streams` consistently from `Server.Config`; expose active/opened/completed/reset stats. Current stats exist at connection level.                                                                                            |
| Request body        | Demand-driven `Server.Body.read`; EOF marks request done; `discard ~drain:true` follows `Drain_up_to n` or resets. Current `Request_body_read` and `Request_body_discard` commands are the model.                                                         |
| Unread bodies       | On early response, apply unread-body policy. Existing tests cover `Drain_up_to 4` waiting for body bytes before responding.                                                                                                                               |
| Response trailers   | Schedule trailers before closing the streaming writer; current implementation has `Response_trailers` and `Response_close`.                                                                                                                               |
| Flow control        | Only pull response chunks when `Body.Writer.flush` resolves. Do not buffer full response streams in memory.                                                                                                                                               |
| Graceful shutdown   | Send GOAWAY, stop admitting new streams, let active streams finish, then close at deadline. Current graceful shutdown waits for active stream completion; extend it to send explicit GOAWAY.                                                              |
| H2 security         | Apply `Eta_http.H2.Security.observe` to server ingress, not only client ingress. The existing security helper already owns cheap frame-envelope policy for SETTINGS/GOAWAY, HPACK block caps, continuation accumulator caps, and header name/value caps.  |

## 3.3 TLS server behavior

Add server-side OpenSSL bindings parallel to the existing client flow:

```ocaml
module Eta_http.Tls.Config.Server : sig
  type t

  val default :
    cert_chain_file:string ->
    private_key_file:string ->
    ?alpn_protocols:string list ->
    ?ca_file:string ->
    ?require_client_cert:bool ->
    unit ->
    t
end

module Eta_http_eio.Tls.Server : sig
  type epoch = {
    alpn_protocol : string option;
    sni : string option;
    peer_certificate_verified : bool;
  }

  val server_of_flow :
    ?host_eio:Eta_eio.Host.t ->
    config:Eta_http.Tls.Config.Server.t ->
    [ Eio.Flow.two_way_ty | Eio.Resource.close_ty ] Eio.Resource.t ->
    (flow * epoch, Eta_http.Server.Error.t) Eta.Effect.t
end
```

The current `Tls_eio` client code already demonstrates the right shape for memory-BIO driving, mutex discipline, handshake completion, and ALPN extraction; do not duplicate policy outside this adapter. 

TLS policy:

| Item               | Target                                                                                |
| ------------------ | ------------------------------------------------------------------------------------- |
| Default ALPN       | `["h2"; "http/1.1"]`. Existing client default already uses this order.                |
| No ALPN            | Route to H1. Existing pure ALPN helper maps `None` to H1.                             |
| Unknown ALPN       | Close connection and increment `alpn_rejected`; do not run an HTTP parser.            |
| Handshake timeout  | Typed `Tls_handshake_failed` or `Request_timeout` at `Accept`/`Connection` layer.     |
| Certificate reload | Not in first production cut; expose restart-based config first. Add hot reload later. |
| SNI                | Start with one certificate; design `sni_callback` as an extension point.              |

## 3.4 Timeouts, limits, and backpressure

Make all production limits explicit in `Config.default`.

| Limit/timeout                      | Default behavior                                                                                                                            |
| ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------- |
| Accept backlog and max connections | Reuse current `backlog`/`max_connections`. Current defaults are 128 backlog and 1024 max connections.                                       |
| Header timeout                     | Close with `408` for H1 if no full request head arrives; H2 connection error or GOAWAY for H2.                                              |
| Header size/count                  | H1 returns `431` or `400` and closes; H2 maps to stream error or connection error depending frame scope.                                    |
| Body size                          | `Server.Body.read_all` already enforces a max; add connection-level body byte accounting so streaming handlers also observe configured max. |
| Idle timeout                       | Close keep-alive H1 connection and idle H2 connection after timeout.                                                                        |
| Handler timeout                    | Optional; when configured, handler failure becomes default error response if headers not committed, otherwise reset/close stream.           |
| Response write timeout             | Close connection or reset H2 stream when peer stops reading.                                                                                |
| Command queue capacity             | Keep bounded; current h2c config has `command_queue_capacity`.                                                                              |
| Backpressure                       | No unbounded body queues. One read demand -> one protocol read scheduling step. One response write flush -> next application chunk.         |

## 3.5 Error mapping

Use this rule: **protocol/transport errors control the connection; handler/application errors become responses only when the response has not been committed.**

| Error                                | H1 wire result                         | H2 wire result                                                  |
| ------------------------------------ | -------------------------------------- | --------------------------------------------------------------- |
| Bad request line/header              | `400` or `431`, `Connection: close`    | stream error if stream scoped; connection error if frame scoped |
| Body too large                       | `413`, drain/close by policy           | `413` response or stream reset by policy                        |
| Request timeout                      | `408`, close                           | stream reset or GOAWAY depending phase                          |
| Handler failure before response      | `500` default response                 | `500` default response                                          |
| Handler failure after response start | close connection or abort chunked body | `RST_STREAM INTERNAL_ERROR`                                     |
| Response write failure               | close                                  | stream reset / connection close                                 |
| TLS handshake failure                | no HTTP response; close                | no HTTP response; close                                         |
| Unsupported ALPN                     | no HTTP response; close                | no HTTP response; close                                         |

# 7. Decisions and tradeoffs

## Reuse

| Decision                                                                 | Reason                                                                                                        | Consequence                                                     |
| ------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------- |
| Reuse current h2c owner-loop model                                       | It already enforces clean ownership: one owner loop, command queue, handler fibers, stats, graceful shutdown. | Faster path to production H2; less protocol-state risk.         |
| Reuse `Server.Request/Response/Body/Error`                               | Backend-neutral server API already exists and tests cover core helpers.                                       | API evolution is incremental rather than a rewrite.             |
| Reuse `Body.Chunked`, header validation, redaction, semconv/tracer/meter | These are already backend-neutral and tested.                                                                 | H1 server can stay small.                                       |
| Reuse pure ALPN state/decision logic                                     | Current `Transport.Alpn` and `Dispatch` already encode `h2`, `http/1.1`, and `None => H1`.                    | Add server-shaped dispatch instead of duplicating ALPN policy.  |
| Reuse OpenSSL memory-BIO pattern                                         | `Tls_eio.client_of_flow` proves the adapter shape.                                                            | Server TLS can mirror client flow ownership and mutex strategy. |

## New abstractions

| New abstraction                          | Why                                                                                                        |
| ---------------------------------------- | ---------------------------------------------------------------------------------------------------------- |
| `H1_server_connection`                   | H1 has very different framing/keep-alive semantics from H2. Mixing it into `Server` would hide ownership.  |
| `Tls_server`                             | TLS handshake, ALPN, and epoch metadata must be explicit and testable.                                     |
| `Server_stats`                           | Current stats are h2c-only; production needs protocol-wide and TLS counters.                               |
| `Eta_http.Server.Config`                 | Separates semantic HTTP limits/timeouts from Eio listener knobs.                                           |
| `H1.Request_parse` / `H1.Response_write` | Existing H1 modules are client/response oriented; server needs request parsing and response serialization. |

## Rejected alternatives

| Alternative                                                 | Rejection reason                                                                                                                 |
| ----------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------- |
| Put Eio server types in `eta_http`                          | Violates package boundary and makes non-Eio backends impossible.                                                                 |
| Require external TLS terminator only                        | Simpler, but does not satisfy HTTPS/TLS/ALPN production server requirement and blocks adversarial h2 TLS tests.                  |
| Implement H1 and H2 through one generic connection module   | The protocol state machines differ too much; generic abstraction would obscure ownership.                                        |
| Enable full HTTP/1.1 pipelining in first production release | Pipelining is rare and complicates response ordering and buffering. Default to no pipelining; add bounded sequential mode later. |
| Auto-compression in the core server                         | Compression changes body accounting and security posture. Keep as an explicit application/transducer layer.                      |
| Add a router/static-file server to core                     | Eta remains a small effect server; route/static helpers can be separate examples or packages.                                    |

# 8. Open questions

| Question                                                                                | Why it matters                                                                                                       |
| --------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- |
| Should production HTTPS support SNI multi-certificate selection in v1?                  | Changes TLS config shape and OpenSSL server callback work.                                                           |
| Should TLS client certificate auth be in v1?                                            | Affects request metadata, TLS config, and error mapping.                                                             |
| Should H1 support bounded sequential pipelining at launch?                              | Affects parser buffering and connection state complexity.                                                            |
| Should h2c upgrade from H1 be supported, or only prior-knowledge h2c?                   | Requires H1 upgrade parser path and protocol handoff.                                                                |
| What are the default max body/header sizes?                                             | Defaults become security posture and interop expectations.                                                           |
| Should handler timeout be enabled by default?                                           | Can surprise long streaming handlers; affects production safety.                                                     |
| Should `runtime_factory` create one runtime per connection, per request, or allow both? | Current h2 code creates a runtime per handler/connection context; production defaults affect overhead and isolation. |
| Should response trailers be allowed for all H1 chunked responses by default?            | Some clients mishandle trailers; config may need compatibility mode.                                                 |
| Should WebSocket server be included in “production HTTP server” v1?                     | There is a WebSocket client; server upgrade support would expand H1 scope.                                           |
| Should stats be monotonic-only plus gauges, or expose histograms/latency?               | Metrics API currently offers counters/gauges; latency may need tracer spans or future meter support.                 |

# Direct recommendations

1. **Do not build TLS or H1 as thin wrappers around h2c.** Keep per-protocol connection owners and share only the backend-neutral `Server` request/response/error/config model.

2. **Make `H1_server_connection` the next implementation milestone.** Without H1, HTTPS ALPN cannot be production-grade because no-ALPN and `http/1.1` clients must route somewhere.

3. **Refactor h2c before adding TLS.** Turn `H2_server_connection.run_h2c` into a generic H2 runner with explicit `Connection_info`; then TLS+h2 is just another flow source.

4. **Add server-side OpenSSL bindings only after H1 and generic H2 are stable.** TLS will otherwise force too many unresolved protocol choices at once.

5. **Promote the testsuite from client interop to server interop.** Add Eta as a server kind, run curl/nghttp against it, and use nginx/caddy as behavior references rather than only as upstream servers.

6. **Make limits and shutdown explicit in the public API.** Production readiness here is less about adding features and more about making every timeout, buffer, body cap, ALPN decision, and close path observable and testable.
