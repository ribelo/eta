Bottom line

The most likely root cause is not H2C protocol logic. It is the TLS wrapper letting both the logical read path and the logical write path drive raw socket reads into the same OpenSSL memory-BIO SSL object. In HTTP/2, the server’s H2 owner loop writes SETTINGS/ACK/response frames while the reader fiber is trying to deliver inbound client frames. If single_write hits SSL_ERROR_WANT_READ and calls feed_bio, the H2 owner/writer side can consume inbound encrypted bytes that should wake the H2 reader path. With no central TLS owner or notification discipline, decrypted/pending plaintext can be stranded until another network event or timeout happens. That matches “simple request may pass, oha-style H2/TLS times out, H2C works, H1/TLS works.” The packaged brief’s observed 58-byte read—preface/settings but not HEADERS—is also consistent with “TLS delivered the first plaintext fragment, then progress depended on a bad scheduling interleaving.”

The key code smell is explicit: tls_eio.ml has one SSL object, raw flow, ssl_mutex, read_mutex, and write_mutex; feed_bio performs raw single_read then BIO_write to OpenSSL’s read BIO, while drain_bio drains OpenSSL’s write BIO to the raw flow. single_read calls feed_bio on WANT_READ, and single_write also calls feed_bio on WANT_READ.

1. Most likely root cause

The most likely root cause is ambiguous raw-read ownership in Tls_eio.Flow_impl.single_write.

A precise failure shape:

H2/TLS connection is established and ALPN selects h2.
The client sends H2 preface/settings and HEADERS, often back-to-back and fragmented differently from the hand-written simple test.
The H2 server owner loop calls drain_writes before and after processing commands. That means it can enter Eio.Flow.write on the TLS flow while the client is also sending request frames. The owner loop does this synchronously before returning to Eio.Stream.take.
Inside TLS single_write, OpenSSL can return SSL_ERROR_WANT_READ. Current code responds by calling feed_bio, which reads encrypted bytes from the raw socket and writes them to OpenSSL’s rbio.
Those bytes may contain application data that the H2 reader fiber should deliver to H2.Server_connection.read. But the writer-side code has no obligation to call SSL_read, enqueue an Ingress, or wake the H2 owner as a reader.
The actual H2 reader can then be blocked on raw socket input even though the TLS object already has bytes buffered or decryptable. The H2 owner waits for commands, and the request times out.

This explains why the issue appears under oha-style H2/TLS behavior: HTTP/2 creates real full-duplex pressure. H1/TLS usually has a much more linear “write request/read response” shape, so this bug is far less likely to manifest.

The “later handshakes stop completing” symptom does not necessarily mean a shared SSL object is corrupted across connections. server_of_flow_with_context creates a fresh server SSL from the shared SSL_CTX per accepted connection. A more likely explanation is that stalled H2/TLS connections remain active and consume run_server connection capacity or scheduler resources, so later clients appear to hang during handshake/accept.

2. Is the current concurrency model valid?

No, not for HTTP/2 over Eio fibers using OpenSSL memory BIOs.

It is partially serialized, but not correctly owned. The current model serializes individual OpenSSL calls with ssl_mutex, and separately serializes raw reads and raw writes with read_mutex and write_mutex. That prevents some memory races, but it does not establish the key protocol invariant:

There must be exactly one owner responsible for feeding inbound ciphertext into rbio and making resulting plaintext visible to the application reader.

Right now, the TLS read path, write path, and handshake path can all drive OpenSSL progress. That is acceptable only for a synchronous blocking wrapper where one operation owns the whole connection until completion. It is risky for Eio full-duplex flows because the write fiber can become a hidden read pump.

The code comment in single_read says reads and writes must proceed concurrently for HTTP/2 and that the shared SSL object is serialized by ssl_mutex; that is directionally right, but it is not sufficient. OpenSSL memory BIO integration needs event ownership, not just a mutex around SSL_read and SSL_write.

3. Should single_write ever read from the underlying raw socket on WANT_READ?

For Eta’s H2/TLS flow: no.

OpenSSL is allowed to return SSL_ERROR_WANT_READ from SSL_write; TLS 1.3 post-handshake messages, key updates, shutdown, and other control traffic can require inbound bytes before a write can complete. But in a full-duplex Eio adapter, single_write should not satisfy that by directly blocking on the raw socket. It should either:

wait until the TLS read owner feeds rbio and signals progress, or
delegate to a single TLS owner/pump fiber that owns all OpenSSL calls and raw socket I/O.

The current single_write branch:

if code = 2 (* SSL_ERROR_WANT_READ *) then (
  feed_bio t;
  write_buf offset length)

is the wrong ownership model for H2/TLS. It lets the writer steal inbound bytes from the reader path.

For a narrow compatibility mode, H1-only one-shot TLS could tolerate writer-side reads because there may not be a concurrent read owner. But the same TLS flow abstraction is being used by H2, so the safe default should be: post-handshake raw reads are owned by the TLS read side or a TLS pump, never by single_write.

4. How ssl_pending should affect single_read

ssl_pending means “decrypted bytes are readable from SSL without touching the network.” The OpenSSL binding exposes it directly.

The invariant should be:

single_read must never block on the raw socket while SSL_pending > 0.

The read algorithm should be:

single_read dst:
  ensure_handshake_done();

  (* Fast path: already-decrypted bytes. *)
  if ssl_pending() > 0:
    return SSL_read(dst)

  (* Also try SSL_read once even when ssl_pending = 0, because rbio may contain
     encrypted records that have not yet been converted into pending plaintext. *)
  rc = SSL_read(dst)

  if rc > 0:
    return rc

  if rc == WANT_WRITE:
    drain_wbio()
    retry

  if rc == WANT_READ:
    drain_wbio()

    (* Re-check before blocking. Another fiber may have fed rbio. *)
    if ssl_pending() > 0:
      retry

    (* Only the read owner may perform this. *)
    read raw socket once
    BIO_write(rbio)
    signal "rbio fed / input progressed"
    retry

Two details matter:

First, SSL_pending only counts plaintext already processed into OpenSSL’s internal application-data buffer. It does not necessarily mean there are no encrypted bytes already sitting in rbio. That is why single_read should still call SSL_read before blocking; SSL_read may process rbio into plaintext without a raw socket read.

Second, if possible, add an OpenSSL binding for SSL_has_pending or rbio pending bytes. SSL_pending is the minimum fix for “do not block when plaintext is already available,” but SSL_has_pending is a better diagnostic and wakeup signal for “OpenSSL still has buffered records to process.”

Also fix Cstruct offsets in the same patch. The current single_read destructures Cstruct.off only for debug display but calls Openssl.read t.ssl storage 0 len; single_write starts from offset 0 inside Cstruct.to_bigarray. If Cstruct.to_bigarray exposes the underlying buffer rather than a shifted view, this corrupts sub-Cstruct reads/writes. Even if it is not the H2/TLS stall root cause, it is too close to the TLS boundary to leave unfixed.

5. Correct ownership model

For the small patch, use this ownership model:

Resource	Owner
Raw socket read side	TLS read path only, via feed_bio; handshake may use it before the flow is exposed
Raw socket write side	TLS write/drain path only, via drain_bio
SSL * object	Serialized by ssl_mutex, but only under the above ownership rules
rbio	Fed only by the raw-read owner
wbio	Drained by drain_bio; may be called after SSL_read, SSL_write, handshake, and shutdown
H2 reader fiber	Owns application plaintext ingress into H2
H2 owner/writer loop	Owns H2 state machine and TLS writes, but must not perform raw reads

For the long-term design, use this ownership model:

Resource	Owner
SSL *, rbio, wbio	One TLS owner/pump fiber
Raw socket read	One raw-reader fiber that reads ciphertext and sends chunks to the TLS owner; it does not touch OpenSSL
Raw socket write	One raw-writer fiber that receives ciphertext from the TLS owner; it does not touch OpenSSL
Application single_read	Waits on a plaintext queue populated by TLS owner
Application single_write	Enqueues plaintext write requests to TLS owner and waits for acceptance/backpressure
Shutdown/close	Sent as commands to TLS owner, which emits close_notify and closes raw flow deterministically

That design avoids “reader path vs writer path” ambiguity completely.

6. Smallest correct fix to try first

The smallest fix I would try is not just “yield and retry” on writer WANT_READ; the prompt says that experiment did not fix it, and it does not establish ownership. The smallest structurally meaningful fix is:

Patch 1: make single_write stop reading the raw socket

In lib/http_eio/tls/tls_eio.ml, replace writer-side WANT_READ handling with a wait for read-side progress.

Current shape:

if code = 2 (* SSL_ERROR_WANT_READ *) then (
  feed_bio t;
  write_buf offset length)

Target shape:

if code = want_read then (
  drain_bio t;
  wait_for_rbio_progress_or_close t;
  write_buf offset length)

Where wait_for_rbio_progress_or_close is signaled by feed_bio after it writes at least one byte into rbio, and by close/shutdown. Do not raw-read from the writer.

Patch 2: make single_read pending-aware before raw blocking

Add a fast path and re-check around WANT_READ:

let pending_plaintext t =
  with_ssl t (fun () -> Openssl.ssl_pending t.ssl)

(* before any feed_bio/raw read *)
if pending_plaintext t > 0 then ssl_read_once ...

(* after WANT_READ, before feed_bio *)
if pending_plaintext t > 0 then retry else feed_bio t

This is mainly to enforce and test the invariant. The current code already attempts SSL_read before feed_bio, but it does not make the “no raw block while pending plaintext exists” invariant explicit, observable, or signaled to waiting writers.

Patch 3: fix Cstruct offsets in TLS read/write

Use the actual Cstruct base offset for OpenSSL reads/writes. Pseudocode:

let { Cstruct.buffer; off; len } = buf in
Openssl.read t.ssl buffer off len

and for writes:

let { Cstruct.buffer; off = base; len } = buf in
let rec write_buf logical_off remaining =
  Openssl.write t.ssl buffer (base + logical_off) remaining
Patch 4: add TLS progress instrumentation under a test flag

Counters:

raw_reads_from_read_path
raw_reads_from_write_path
ssl_read_want_read
ssl_write_want_read
ssl_pending_before_raw_read
rbio_feed_notifications
writer_waited_for_rbio

The acceptance criterion for H2/TLS should be:

raw_reads_from_write_path = 0
ssl_pending_before_raw_read = 0
all H2/TLS requests complete
Why this is the smallest correct fix

It preserves the existing flow API and does not require rewriting H2. It changes only the ownership contract inside tls_eio.ml. It also directly targets the dangerous behavior: writer-side raw reads.

7. Long-term design if the small fix is risky

The long-term design should be a single TLS owner/pump, or at least a single OpenSSL owner plus separate raw read/write workers that never touch OpenSSL.

A good architecture:

application read fiber
  -> waits on plaintext queue

application write fiber
  -> sends Write_plaintext(bytes, ack) command

raw reader fiber
  -> reads ciphertext from socket
  -> sends Inbound_ciphertext(bytes) command

raw writer fiber
  <- receives outbound ciphertext chunks
  -> writes to socket

TLS owner fiber
  owns SSL/rbio/wbio exclusively
  handles:
    Inbound_ciphertext -> BIO_write(rbio), SSL_read until WANT_READ, enqueue plaintext
    Write_plaintext    -> SSL_write, drain wbio, ack after accepted/backpressured
    Shutdown           -> SSL_shutdown, drain wbio, close

The owner fiber is the only place that calls:

Openssl.handshake
Openssl.read
Openssl.write
Openssl.shutdown
Openssl.bio_read
Openssl.bio_write
Openssl.bio_write_pending
Openssl.ssl_pending

This is cleaner than trying to reason about nested ssl_mutex, read_mutex, write_mutex, conditions, and operation interleavings. The low-level OpenSSL binding is documented as “No I/O — pure state-machine driver,” which is exactly the shape that wants an explicit state-machine owner above it.

API breakage is acceptable here. For a public edge server, correctness beats preserving a leaky TLS-flow abstraction.

9. Could H2 writer/connection also contribute?

Yes, but as an amplifier rather than the primary root cause.

The H2 server owner loop is structured as:

drain_writes t;
let command = Eio.Stream.take t.commands;
handle_command t command;
...
owner_loop t

and handle_command drains writes again after ingress and response operations. That means if TLS single_write blocks waiting for raw input, the H2 owner is blocked too. It cannot process already queued ingress or response commands. Under H2/TLS, this is much more likely than under H2C because writes pass through OpenSSL and can produce WANT_READ/WANT_WRITE.

The H2 writer coalescing change helps only by changing fragmentation and timing. H2.Writer.write_iovecs now coalesces H2 iovecs into one Cstruct and calls Eio.Flow.write once. That may make simple tests pass more often, but HTTP/2 over TLS must tolerate arbitrary TLS record and frame fragmentation. Coalescing is a performance/timing improvement, not a correctness proof.

So the evidence still points primarily to TLS:

H2C works under load
H1/TLS works
H2/TLS simple cases may pass
H2/TLS under full-duplex/multiplexed client behavior stalls

That is the signature of a TLS full-duplex scheduling bug, not an H2 frame parser bug.

TLS wrapper invariants to enforce

Use these as code comments and tests:

Single rbio feeder after handshake: only the TLS read owner may read from raw socket and BIO_write into rbio.
No writer-side raw reads: single_write must not call raw single_read.
No raw read while plaintext pending: single_read must not block on raw socket while SSL_pending > 0.
OpenSSL calls are serialized: no concurrent SSL_read, SSL_write, SSL_shutdown, BIO_read, or BIO_write on the same SSL.
Do not hold ssl_mutex across blocking raw I/O.
Do not hold raw read/write locks across arbitrary OpenSSL progress loops longer than necessary.
Every OpenSSL operation that may generate wbio bytes is followed by drain_bio.
Every successful rbio feed wakes any writer waiting on WANT_READ.
TLS app bytes consumed by the TLS layer must become visible to exactly one application reader.
Connection close resolves all waiters: read waiters, write waiters, handshake waiters, and shutdown waiters.
